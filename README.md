
### Техническое задание

Современные веб-страницы часто выполняют сложные вычисления (например, фильтрация изображений, анализ данных, 3D-графика). Если выполнять их в основном потоке браузера, интерфейс "зависнет". Для решения этой проблемы существуют Web Workers — фоновые процессы, которым основной поток может отправлять задачи и получать от них результаты, не блокируя пользовательский интерфейс.

В этой работе смоделируем именно такую систему:

* **Browser.exe** — управляющий процесс, который играет роль диспетчера задач для браузера. Он отвечает за распределение работы между фоновыми процессами.
* **Worker.exe** — фоновый вычислительный процесс (Web Worker), который получает задачи, "тяжело" работает и отправляет результат обратно.

Взаимодействие между ними осуществляется через именованные каналы.

#### Роль программы Browser.exe ("Сервер")

1. При запуске запрашивает у пользователя:
* N — количество Worker-процессов (пул).
* M — общее количество задач.


2. Для каждого Worker-а (от 0 до N-1) создает два именованных канала:
* Для отправки задач: `\\.\pipe\worker_in_{ID}`
* Для получения результатов: `\\.\pipe\worker_out_{ID}`


3. Запускает N экземпляров `Worker.exe`, передавая ID через аргументы командной строки.
4. Распределяет задачи: отправляет задачу свободному воркеру, дожидается ответа, затем отправляет следующую.
5. Протокол обмена:
* Browser записывает структуру задачи в канал `worker_in`.
* Browser блокируется в ожидании ответа из канала `worker_out`.


6. После выполнения всех задач отправляет "команду-завершения" (`TASK_EXIT`).
7. Дожидается завершения всех Worker.exe, закрывает дескрипторы и завершает работу.

#### Роль программы Worker.exe ("Клиент")

1. Получает свой ID из аргументов командной строки.
2. Подключается к каналам `worker_in_{ID}` и `worker_out_{ID}`.
3. Основной цикл:
* Ожидает и читает задачу из входного канала.
* Если тип задачи `TASK_EXIT` — выходит из цикла.
* Выполняет вычисления согласно варианту (**Преобразование Фурье**).
* Отправляет структуру с результатом обратно через `worker_out`.


4. Корректно закрывает дескрипторы и завершает работу.

### Тестирование

**Группа 1: Позитивные тесты**

* **1.1 Один воркер, одна задача**: Проверка базовой логики и корректности вычислений.
* **1.2 Пул воркеров, много задач**: Проверка распределения нагрузки и синхронизации.
* **1.3 Корректное завершение**: Убедиться, что процессы не "виснут" в памяти после TASK_EXIT.

**Группа 2: Граничные тесты**

* **2.1 Пустая задача**: Отправка пустого массива данных. Worker должен вернуть пустой результат и не упасть.
